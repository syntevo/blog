# SmartGit Overlap

Welcome our new feature in SmartGit 26.1, [**Overlap**](https://docs.syntevo.com/SmartGit/Latest/Manual/GUI/Overlap).

[![](/assets/images/overlap_icons.png)](/assets/images/overlap_icons.png)

Many of our features have a somewhat direct counterpart in the git cli. Some stray a bit from the trodden path, such as the [**Graph View**](https://docs.syntevo.com/SmartGit/Latest/Manual/GUI/Graph-View) or the  [**Interactive Rebase**](https://docs.syntevo.com/SmartGit/Latest/Manual/GUI/Branch/Rebase-Interactive), but they do not try to hide the fact that they share a lot with their character based siblings.

**Overlap** is one of the exceptions to this rule, and that immediately presents a problem: What is this thing no one knows anything about, why is it here, and how do I get back my precious horizontal screen real estate?

Hang on for the answer to the last question, but hopefully I can convice you to give Overlap a try.

[![](/assets/images/overlap_tooltips.webp)](/assets/images/overlap_tooltips.webp)

## Thinking outside the DAG

In a version control system based on patches (I'm thinking of [**Pijul**](https://pijul.org/), there may be others), this information is easy to come by, because changes are what's being tracked. Contrast that with git, which tracks the contents instead. That's why certain operations in git which are properties of the underlying data model - the git DAG - are fast and efficient. Other operations, such as rebasing commits, are not properties of this model, therefore aren't implemented as efficiently, and as a consequence are harder to come by.

[![](/assets/images/overlap_fix_selection.webp)](/assets/images/overlap_fix_selection.webp)

## git help overlap

**Overlap** is a small, fixed-size column located before the commit message column in the Graph View, built to add another dimension to the Graph View. Where the graph displays the dependencies between commits based on the git DAG (i.e. the flow of commits and their ancestors), Overlap displays dependencies between commits based on _changes_.

Let's look at this using a simple example.

Consider a repository with two files, A and B, and a number of commits with the following modifications:

1. Modify line #3 in A
2. Delete line #1 in B
3. Modify line #7 in A
4. Delete file B

The Graph View (or `git log`) tells you many important things, most importantly it displays the _commits_ (including metadata such as authorship) and their _order_.

What it doesn't do is showing you if (and how) the changes (i.e. the modifications of files) in the commits relate to each other.

This is where **Overlap** comes into play. For a selected commit, it shows you how the other commits relate to your selection.

Let's say we select commit #3 - Overlap shows you the following information, encoded in a set of small colored icons:

1. Strong Overlap
2. No Overlap
3. Selected Commit
4. No Overlap

At first glance, you can see that the changes in commit #2 and #4 have no relation to your selection at all, meaning that there are no common files modified at all.

Likewise, it tells you immediately that commit #1 has a strong overlap with your selected commit - they both modify file A.

It even tells you that your selected commit and commit #1 likely can be re-ordered, because if they couldn't (because they modify the same lines of code), Overlap would show you another icon indicating shared modifications.

## A flashlight for exploring branches

This feature being quite new means we ourselves are only using it for a while now, and as such we surely haven't come across every usecase just yet. But to name a few we've discovered so far:

* Quickly getting a feel for a branch
    * Which are the most interesting commits
    * Are the changes separated, or is it more of a zig-zag across files?
* Detecting potential problems
    * Are there changes in commits which should be split off?
    * Do some commits modify exactly the same files and should be squashed together?
    * Can the commits be re-ordered to strengthen their story?
* As a helper when cleaning up
    * When re-ordering commits, where can I re-order without getting conflicts?
    * When splitting off modifications, where do the changes probably belong?
* For quickly spotting conflicts
    * The tooltip shows potentially conflicting files without requiring another step 

There's probably plenty of other uses. Have you found others? [Tell us!](https://www.reddit.com/r/SmartGit)

## Getting rid of it

Go to `Hamburger (the three horizontal lines in the top right corner of the Graph) | Columns | Overlap`, or set the low-level property `log.graph.overlap.enabled` to completely remove Overlap, even from that menu.
